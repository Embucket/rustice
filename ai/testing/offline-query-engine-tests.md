# Offline query-engine tests

## Purpose

Offline query-engine tests prove Rustice behavior without Snowflake credentials, Snowflake network access, or an external Snowflake service. They are the default development loop for parser, planner, function, catalog, execution, and local REST changes.

The goal is to make local execution trustworthy while still leaving room for separate Snowflake oracle tests when exact Snowflake behavior must be checked.

## What to test without Snowflake

Test these locally first:

- SQL parsing and AST rewrites in `executor` and `functions::visitors`
- DataFusion planning, logical analyzers, optimizers, physical optimizers, and type planning
- scalar, aggregate, window, and table function behavior
- type coercion, casts, null handling, date/time/timestamp behavior, and result formatting
- DDL/DML behavior that the local executor owns, including local writes where supported
- catalog discovery, information schema, schema/table/view resolution, and case normalization
- local COPY/MERGE/table behavior with memory, temp file, or file-backed catalog fixtures
- REST response shape using the local `api-snowflake-rest` test server
- session/JWT/cookie/SPCS-ingress extraction behavior that can be exercised with local headers

Unsupported behavior should fail explicitly in local tests. Do not turn a local failure into assumed Snowflake fallback unless a routing test and policy exist.

## Preferred fixtures

Prefer the smallest fixture that exercises the behavior:

- In-memory DataFusion sessions from `crates/functions/src/tests/utils.rs` for function-only tests.
- `executor::test_helpers::create_df_session()` or `create_df_session_with_catalog_url("/dev")` for query-engine tests.
- Temporary directories/files via `tempfile` for local file catalog and COPY INTO coverage.
- Small inline `VALUES` tables or setup queries in the local `test_query!` macros.
- Local Parquet/Iceberg output generated by the test itself for planner and scan behavior.
- REST tests using `crates/api-snowflake-rest/src/tests/create_test_server.rs`, which starts a local server against a dev catalog.
- Existing snapshots under crate-local `snapshots/` directories for stable expected output.

Avoid shared mutable fixtures unless the test is specifically about shared state. Avoid networked S3 or remote REST catalog fixtures in the default loop; existing network-sensitive tests are ignored by default.

## Choosing the narrowest test

Start with the crate that owns the behavior:

- Function implementation: `cargo test -p functions -- <function_filter>`
- Function SQL integration or planner interaction: `cargo test -p executor -- <query_filter>`
- Parser/planner rewrite: `cargo test -p executor -- <command_or_analyzer_filter>`
- Catalog discovery: `cargo test -p catalog -- information_schema`
- Metastore model/object-store behavior: `cargo test -p catalog-metastore`
- REST wire shape: `cargo test -p api-snowflake-rest test_rest_api -- --test-threads=1`
- Session/auth extraction: `cargo test -p api-snowflake-rest-sessions`
- Compatibility corpus path: `cargo test -p embucket-sqllogictest -- <path-substring> --test-threads 1`

Use wider commands after the focused test passes:

```bash
cargo test -p executor
cargo test -p functions
cargo test -p api-snowflake-rest -- --test-threads=1
cargo test --workspace --all-targets --exclude api-snowflake-rest --exclude embucket-sqllogictest
```

The root `Makefile` target `make test` currently wraps `cargo test`. Prefer explicit `cargo test -p ...` commands when documenting or reporting verification, because they make the tested crate clear.

## Limitations of offline tests

Offline tests cannot prove:

- exact live Snowflake semantics for every SQL edge case
- remote Snowflake authorization, account, warehouse, governance, grants, or stage behavior
- remote network behavior, latency, retry behavior, or remote error wording
- a passthrough/fallback path that is not implemented and exercised locally
- that a local Iceberg/catalog view matches a live Snowflake account

Offline tests can still define Rustice's current expected behavior. When local behavior intentionally differs, document the difference and keep a local regression test.

## When Snowflake or oracle testing is required

Use Snowflake-backed oracle testing as an additional layer when:

- exact Snowflake semantics are unknown and cannot be inferred from local tests
- a routing policy depends on whether a statement should be passthrough/fallback eligible
- client behavior depends on live Snowflake error codes, messages, or metadata
- remote account/session/warehouse/grant behavior is the feature under test
- the change implements actual passthrough/fallback/hybrid execution against Snowflake

Oracle results should feed back into local tests whenever possible. Snowflake access must not be required for every small local engine or compatibility change.
